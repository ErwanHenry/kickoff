<objective>
Create public match page accessible via WhatsApp shareable link — implementing zero-friction guest RSVP with httpOnly cookie persistence, waitlist management, and race-condition prevention.

Purpose: This is the core differentiator feature — guests can RSVP without creating an account. The page must load fast (SHARE-03 <1s on 3G), handle race conditions during RSVP (two guests competing for last spot), persist guest identity via httpOnly cookies (GUEST-03), enable cancellation (GUEST-05), and implement the complete waitlist status flow (WAIT-03, WAIT-04). The RSVPButton state machine must handle all guest states: initial, confirmed, waitlisted, and return visit detection (GUEST-04).

Output: Working public match page at /m/{shareToken} with guest RSVP, cookie-based identity, cancellation, automatic waitlist, position tracking, and complete status transitions.

  <files>src/app/api/rsvp/route.ts</files>

  <action>
Create src/app/api/rsvp/route.ts with:

1. Add "use server" directive
2. Import: db, matches, matchPlayers schema, eq, sql, desc from drizzle-orm, generateGuestToken, setGuestToken, revalidatePath, z
3. Create rsvpSchema: z.object({ shareToken: z.string().length(10), guestName: z.string().min(1).max(50) })
4. Export async function rsvpMatch(formData: FormData):
   - Parse input: shareToken from formData, guestName from formData
   - Validate with rsvpSchema
   - Generate guestToken via generateGuestToken()
   - Get match by shareToken (return error if not found)
   - Check match status (return error if locked/played/rated)
   - **CRITICAL: Use db.transaction() to prevent race conditions**:
     * Lock match row with .for("update")
     * Count confirmed players in transaction
     * Determine if full: confirmedCount >= match.maxPlayers
     * If full: insert with status="waitlisted", calculate position
     * If not full: insert with status="confirmed", update match.status to "full" if now full
   - Set guestToken via setGuestToken()
   - RevalidatePath(`/m/${shareToken}`)
   - Return { success, guestToken, status, waitlistPosition }

5. Export async function cancelRsvp(formData: FormData):
   - Parse shareToken and guestToken from formData
   - Get match and player by guestToken
   - Update player.status to "cancelled"
   - Set cancelledAt timestamp
   - Check if match was full and has waitlist:
     * If yes, change match.status from "full" to "open" (WAIT-03)
   - RevalidatePath(`/m/${shareToken}`)
   - Return { success }

   **CONCURRENT SCENARIO HANDLING (Critical for GUEST-05):**
   The transaction in rsvpMatch uses FOR UPDATE locking which serializes access. When a cancellation happens simultaneously with a new RSVP:
   - If cancellation commits first → spot opens → new RSVP gets confirmed
   - If new RSVP commits first → full → cancellation opens spot (WAIT-03 satisfied)
   - Database ACID guarantees ensure no lost updates

   Per RESEARCH.md "Pitfall 1: Race Conditions in RSVP" with FOR UPDATE locking.
  </action>
